# S3 API Performance: 파일 I/O 대체재로서의 S3 vs EBS vs EFS 심층 리서치

> 조사일: 2025년 기준 | 출처: AWS 공식 문서, 벤치마크 블로그, Hacker News 토론 등

---

## 1. S3 API 성능 특성

### 1.1 GetObject/PutObject 지연시간

| 항목 | 수치 | 비고 |
|------|------|------|
| S3 Standard TTFB (일반) | **100~200ms** | EC2 c6g.4xlarge, 10GbE 환경 (2025) |
| S3 Standard p99 tail latency | **100ms+** | 검색 워크로드 벤치마크 (nixiesearch, 2025) |
| S3 Express One Zone TTFB | **한 자릿수 ms** | S3 Standard 대비 5~10배 빠름 |
| S3 Express p99 (WarpStream 벤치마크) | **169ms** (produce), median **105ms** | 비교 대상 대비 3배 낮음 (2024) |

- 오브젝트 크기별 세분화된 레이턴시 벤치마크는 공개 자료에서 찾기 어려움
- 집계 처리량: 업로드 1,444 MB/s, 다운로드 1,816 MB/s, 소형 객체 319 ops/sec (rabata.io, 2025)

### 1.2 S3 Transfer Acceleration

- **장거리/크로스 리전 전송**에 주로 효과적
- 2026년 HTTP/3, QUIC 지원으로 레이턴시 30% 감소, 글로벌 사용자 업로드 60% 빨라짐
- **동일 리전 내 프로그래밍 API 접근에는 효과 없음**

### 1.3 Multipart Upload

- 대용량 파일에서 단일 PUT 대비 **20% 빠름** (2026 최적화 API 기준)
- AWS 권장: 대용량 요청 시 병렬화 + AWS CRT 라이브러리 사용
- 최적 파트 크기: 명시적 수치 없으나, 8+ vCPU / 32GB RAM 인스턴스에서 20% 성능 향상
- 프리픽스당 3,500 PUT/COPY/POST/DELETE per second 확장

### 1.4 Range GET (Byte-Range Fetch)

- 레이턴시 오버헤드에 대한 구체적 벤치마크 부재
- AWS Storage Lens로 읽기 요청 크기 분포 모니터링 가능
- 소형 Range GET이 성능 저하 유발 시 배치 처리 또는 S3 Express 권장

## 2. S3 vs EBS vs EFS 성능 비교

### 2.1 IOPS 비교

| 스토리지 | IOPS | 비고 |
|----------|------|------|
| **S3** | 5,500 GET/s + 3,500 PUT/s (프리픽스당) | 요청 기반, 프리픽스 분산으로 선형 확장 |
| **EBS gp3** | 3,000 기본 → **최대 80,000** | 2025년 9월 확장 (기존 16,000) |
| **EFS** | 처리량 모드에 따라 가변 | 분산 파일 접근에 최적 |

### 2.2 처리량 (MB/s) 비교

| 스토리지 | 처리량 | 비고 |
|----------|--------|------|
| **S3** | 집계 처리량 높음 (병렬화에 비례) | 단일 객체 MB/s 보장 없음 |
| **EBS gp3** | 125 MB/s 기본 → **최대 2,000 MB/s** | 2025년 확장 (기존 1,000 MB/s) |
| **EFS** | Burst/Provisioned 모드, 수천 MB/s (클라이언트 합산) | Max I/O 모드에서 분산 순차 워크로드에 적합 |

### 2.3 레이턴시 비교

| 스토리지 | 레이턴시 | 적합 워크로드 |
|----------|----------|--------------|
| **EBS gp3** | **서브 ms ~ 한 자릿수 ms** | DB, 랜덤 I/O |
| **S3 Standard** | **한 자릿수 ~ 두 자릿수 ms** | 객체 기반 순차 접근 |
| **S3 Express One Zone** | **한 자릿수 ms** | 저지연 객체 접근 |
| **EFS** | **낮은 ms (mid-ms)** | 공유 파일 시스템 |

### 2.4 랜덤 접근 패턴

- **EBS gp3**: 4KB 랜덤 읽기에 최적, 최대 80,000 IOPS → **DB 워크로드의 유일한 선택지**
- **S3**: 소형 객체 랜덤 접근 시 성능 매우 나쁨 (프리픽스당 5,500 GET/s, 전체 객체 페치 필요, 높은 레이턴시)
- **EFS**: S3보다 낫지만 EBS보다 레이턴시 높음

### 2.5 핵심 판단

> **EBS = 최저 레이턴시 + 최고 IOPS (DB 필수)**
> **EFS = 분산 처리량 최적 (다중 인스턴스 공유)**
> **S3 = 집계 처리량 최고 (대용량 순차 접근, 비용 효율)**

## 3. S3를 애플리케이션 데이터 스토어로 사용

### 3.1 S3 Mountpoint for Amazon S3

FUSE 기반 오픈소스 파일 클라이언트로 S3 버킷을 로컬 파일시스템처럼 마운트.

**지원/미지원 현황 (2025년 11월 기준):**

| 작업 | 지원 여부 |
|------|-----------|
| 순차 읽기 | O (고성능) |
| 랜덤 읽기 | O |
| 순차 쓰기 | 제한적 |
| **랜덤 쓰기** | **X** |
| **Append** | **X** |
| **Rename** | **X** |
| **Symlinks** | **X** |
| **Hard links** | **X** |
| **파일 락킹** | **X** |

**주요 업데이트:**
- 2024.11: S3 Express One Zone 기반 **공유 읽기 캐시** → S3 Standard 대비 **최대 7배** 빠른 후속 읽기
- 2025.08: CSI 드라이버 v2, 캐싱 성능 향상 (금융 시뮬레이션에서 **2배 빠름**), SELinux 지원
- 2025.11: Amazon Linux 2023에 통합, OpenTelemetry 기반 모니터링

**결론**: ML 학습 데이터, 게노믹스 등 **대규모 순차 읽기 워크로드**에 적합. 범용 파일시스템 대체 불가.

### 3.2 S3 Object Lambda

- GetObject 요청을 Lambda 함수를 통해 라우팅, 실시간 데이터 변환
- **레이턴시 오버헤드 큼**: Lambda 호출 비용 + 콜드 스타트, 최대 60초 타임아웃
- 용도: PII 마스킹, 이미지 리사이징, 포맷 변환 (CSV→JSON)
- **서브 ms 요구에는 부적합**, 핫 객체는 사전 처리 권장

### 3.3 S3 Access Points

- 버킷당 수천 개의 엔드포인트 생성, 각각 독립적 IAM 정책
- VPC 전용 접근, 멀티테넌트 앱에서 정책 관리 단순화
- Object Lambda와 통합하여 변환 라우팅

### 3.4 실제 사용 패턴

**성공하는 패턴:**
- ML 데이터셋, 로그, 백업 (순차 대용량 처리)
- Object Lambda + Mountpoint 조합의 분석 파이프라인

**실패하는 패턴:**
- DB, 실시간 ML, 게임 상태 관리 등 **랜덤 소형 I/O**
- S3의 10~100ms 객체별 레이턴시가 성능 병목

## 4. 비용 비교 (us-east-1, 2024-2025 가격)

### 4.1 GB당 월간 스토리지 비용

| 스토리지 클래스 | $/GB/월 | 비고 |
|----------------|---------|------|
| S3 Standard | **$0.023** | 첫 50TB |
| S3 Express One Zone | **$0.11** | 2025.04 가격 인하 후 (기존 대비 31% ↓) |
| S3 Intelligent-Tiering | $0.023 ~ $0.0125 | 접근 패턴에 따라 자동 이동 |
| EBS gp3 | **$0.08** | |
| EFS Standard | **$0.30** | 리전/멀티AZ |
| EFS IA | **$0.025** | |
| EFS One Zone | $0.16 | |
| EFS One Zone IA | $0.0133 | |

### 4.2 요청 비용

| 작업 | S3 Standard | S3 Express One Zone |
|------|-------------|---------------------|
| PUT (1,000건당) | $0.005 | $0.0025 (50% ↓) |
| GET (1,000건당) | $0.0004 | $0.0002 (85% ↓ from 원래 가격) |

EBS, EFS: 요청당 비용 없음 (스토리지 + IOPS/처리량 프로비저닝 비용)

### 4.3 EBS 대비 S3 손익분기점

S3 Standard ($0.023/GB) vs EBS gp3 ($0.08/GB), 차액 = $0.057/GB/월

| 요청 유형 | 손익분기 요청수/GB/월 | 해석 |
|-----------|----------------------|------|
| 전체 GET | **1.425억 건** | 비현실적 — S3가 훨씬 저렴 |
| 전체 PUT | **1,140만 건** | 여전히 높음 |
| GET/PUT 50:50 | **~2,000만 건** | S3가 일반적으로 저렴 |

> **결론: 극단적인 쓰기 워크로드가 아닌 한 S3가 비용 효율적**

### 4.4 100GB + 100만 읽기/월 총비용 예시

| 서비스 | 스토리지 | 요청 비용 | **월 총비용** |
|--------|----------|-----------|--------------|
| S3 Standard | $2.30 | $0.40 | **$2.70** |
| EBS gp3 | $8.00 | $0 | **$8.00** |
| EFS Standard | $30.00 | $0 | **$30.00** |
| EFS IA | $2.50 | ~$1 (검색비) | **~$3.50** |

### 4.5 데이터 전송 비용

- **S3 → EC2 (동일 리전)**: 무료
- **EBS**: 무료 (단일 인스턴스 연결)
- **EFS**: 동일 AZ 무료, **크로스 AZ 유료** (멀티 AZ 워크로드 시 비용 증가)

## 5. 최신 S3 기능 (2023-2025)

### 5.1 S3 Express One Zone (2023년 re:Invent 발표)

- **한 자릿수 ms 레이턴시**, S3 Standard 대비 최대 **10배 빠름**
- 초당 수십만 요청 처리, 분당 수백만 요청 가능
- **디렉터리 버킷** 사용: 단일 AZ 배치, 세션 기반 인증 (5분 토큰)
- 스토리지: $0.11/GB/월 (S3 Standard의 ~4.8배이나 요청 비용 50% 저렴)
- 2025.04: 스토리지 31%, PUT 55%, GET 85% 가격 인하

### 5.2 S3 Conditional Writes

| 날짜 | 기능 |
|------|------|
| **2024.08.20** | `If-None-Match` 헤더 지원 — 객체 미존재 시에만 생성 (PutObject, CompleteMultipartUpload) |
| **2024.11.25** | `If-Match` 헤더 추가 — ETag 기반 **Compare-and-Swap (CAS)** 패턴 지원 |
| 2024.11.25 | 버킷 정책에서 `s3:if-none-match`, `s3:if-match` 조건 키로 강제 적용 가능 |

- 추가 비용 없음, 모든 AWS 리전에서 SDK/API/CLI로 사용 가능
- **분산 시스템의 동시성 제어에 핵심적인 진전** — 외부 락 매니저 없이 낙관적 동시성 구현 가능

### 5.3 S3 Express One Zone Append (디렉터리 버킷)

- S3 Express One Zone 디렉터리 버킷에서 기존 객체에 데이터 **추가(append)** 가능
- 최대 10,000 파트 (각 파트 최대 5GB, 최소 크기 제한 없음)
- **단일 AZ, 높은 스토리지 비용** ($0.11 vs $0.023)
- 동시 append 시 외부 조율 필요 (예: DynamoDB 락 + HeadObject 오프셋 확인)
- CopyObject로 파트 카운트 리셋 가능

### 5.4 Mountpoint 업데이트 요약

- 2024.11: S3 Express 기반 공유 캐시 (7배 빠른 읽기)
- 2025.08: CSI 드라이버 v2, SELinux
- 2025.11: Amazon Linux 2023 통합, OpenTelemetry 모니터링

## 6. S3의 한계 (파일 I/O 대체 시)

### 6.1 Append 불가 (Standard)

- S3 Standard에서는 **객체 전체를 다시 써야 함** (no native append)
- S3 Express One Zone 디렉터리 버킷에서만 append 지원 (비용 4.8배)
- 워크라운드: Multipart Upload 패턴, 프리픽스 샤딩 (예: 시간별 로그 분할)

### 6.2 파일 락킹 없음

- EFS: POSIX 파일 락 지원, EBS: 단일 인스턴스 배타적 연결
- S3: **네이티브 락 없음** → DynamoDB 기반 외부 락 구현 필요
- Conditional Writes (If-Match) → **단일 작성자 또는 CAS 패턴에서는 락 대체 가능**
- 다중 작성자 append 시나리오에서는 여전히 외부 조율 필수

### 6.3 객체 메타데이터 제한

- 사용자 메타데이터: **최대 ~2KB**, 약 10개 키-값 쌍 (공식 문서 확인 필요)
- 시스템 메타데이터 별도

### 6.4 일관성 모델

- **2020년 12월**: S3 강한 일관성(strong consistency)으로 전환
- 모든 작업 (PUT, GET, DELETE, **LIST 포함**) 강한 일관성 보장
- 이전: 덮어쓰기/LIST는 최종 일관성(eventual consistency)이었음

### 6.5 용량 제한

| 스토리지 | 최대 크기 |
|----------|-----------|
| S3 객체 | **5TB** |
| EBS 볼륨 | **64TB** |
| EFS | **제한 없음** (자동 확장) |

### 6.6 요청 속도 제한 및 스로틀링

- 프리픽스당: 3,500 PUT + 5,500 GET per second
- 초과 시: **503 Slow Down** 에러 → S3가 자동으로 점진 확장
- 프리픽스 분산으로 선형 확장 가능 (10개 프리픽스 → 55,000 GET/s)
- 프리픽스 수 제한 없음

### 6.7 최소 과금 크기

- S3 Standard: 객체당 최소 **128KB** 과금 (실제 크기가 작더라도)
- Multipart: 파트 최소 **5MB** 권장 (마지막 파트 제외)

## 7. 종합 판단: S3를 파일 I/O 대체재로 쓸 수 있는가?

### 적합한 경우

| 시나리오 | 이유 |
|----------|------|
| 대용량 순차 읽기/쓰기 (ML, 미디어, 로그) | 높은 집계 처리량, 저렴한 비용 |
| Write-once-read-many 패턴 | 객체 불변성이 오히려 장점 |
| 이벤트 드리븐 처리 (S3 Event → Lambda) | 네이티브 통합 |
| 분석 데이터 레이크 | Parquet/ORC + S3 Select/Athena |

### 부적합한 경우

| 시나리오 | 이유 |
|----------|------|
| DB 워크로드 (랜덤 4KB I/O) | 레이턴시 100ms+ vs EBS 서브ms |
| 실시간 앱 상태 관리 | 락 없음, append 제한 |
| 빈번한 소형 객체 업데이트 | 전체 객체 재작성 필요, 128KB 최소 과금 |
| POSIX 호환 필요 (rename, symlink 등) | 미지원 |

### 하이브리드 아키텍처 권장

```
Hot Data (서브ms 필요)  →  EBS gp3
Shared File Access      →  EFS
Warm/Cold Object Access →  S3 Standard
Low-Latency Objects     →  S3 Express One Zone
Archive                 →  S3 Glacier
```
